# AI Agent Event Planner

## Overview
The **AI Agent Event Planner** is a LangGraph and LangChain-based application that automates event content generation for youth and church events. The system takes structured event details as input and produces a proposal email, a standardized registration form, a QR code, and a promotional flyer through a multi-step AI workflow.

---

## Project Goals
This application helps automate repetitive event-planning tasks by generating communication and registration materials from a single event input payload. Instead of manually drafting each artifact, the system orchestrates AI and utility functions to create a full event package.

Typical outputs include:
- A proposal email for leadership or stakeholders
- A youth registration form schema
- A QR code linked to the registration form which is hosted on streamlit community cloud
- A flyer image for promotion

## app.py Purpose
`app.py` is the core workflow module for the application. It defines the shared graph state, utility functions, AI-driven generation steps, and orchestration logic needed to transform event details into multiple usable outputs.

Its responsibilities include:
- managing the multi-step workflow state
- generating standardized form schemas
- formatting event metadata
- creating QR codes
- coordinating flyer design and rendering
- supporting downstream use by the Streamlit apps

In practice, `app.py` acts as the business-logic layer of the system.

---

## Architecture Overview
The application uses a **graph-based architecture** rather than a single procedural script. This makes it easier to break the workflow into reusable nodes and preserve context between steps.

### High-level workflow
1. User enters event details in `ui_app.py`
2. `run_task.py` receives the selected task and payload
3. A LangGraph pipeline is created using nodes from `app.py`
4. Shared state is passed from one node to the next
5. The system generates one or more output artifacts
6. The UI displays generated text and downloadable files

---

### Primary input
The main input is a dictionary stored under `event_details`, typically containing:
- `event_name`
- `event_date`
- `event_time`
- `location`
- `form_url`
- optional fields like description, contact email, participant limits, and age range

### Derived outputs
From that input, the application generates:
- email text
- form JSON
- QR code image
- flyer image

### Internal file-based integration
The form schema generated by `app.py` is saved to `current_event_form.json`, which is then automatically loaded by `form_app.py`. This creates a practical handoff between generation logic and the registration UI.

---

## AI Orchestration Design
This project uses both **LangGraph** and **LangChain** to manage AI behavior.

### LangGraph role
LangGraph is used to define workflow steps as nodes and route execution from one stage to another. This is especially useful for tasks like event package generation where several outputs depend on the same shared context.

### LangChain role
LangChain is used for agent creation and tool-oriented generation. In the flyer workflow, the code creates specialized agents such as a flyer designer and an editor, each with distinct responsibilities.

The application includes conversation-state handling through the `messages` field in `State`.

## ui_app.py Role
`ui_app.py` is the main Streamlit interface for the application.

### User workflow
1. Enter event name, date, time, and location
2. Add the registration form URL
3. Optionally include description and contact email
4. Click **Generate Content**
5. View the AI-generated output and download resulting files

The interface is designed to make the app accessible to end users who do not need to interact with the graph or prompt logic directly.

---

## form_app.py Role
`form_app.py` is the registration form interface. It reads the generated `current_event_form.json` file and renders a standardized event form dynamically.

### Key behavior
- auto-loads form schema from disk
- falls back to a default schema if no generated form is found
- supports event-specific rendering without manual form editing
- includes an admin password setting through environment variables

This file shows how `app.py` integrates with a downstream application rather than stopping at raw model output.

## Example Usage
Below is an example payload used by the system:

```python
payload = {
    "event_details": {
        "event_name": "Youth Basketball Clinic",
        "event_date": "2026-04-15",
        "event_time": "3:00 PM - 5:00 PM",
        "location": "City Sports Complex",
        "form_url": "https://forms.example.com/register",
        "description": "Join us for an exciting youth basketball clinic!",
        "contact_email": "pastor@church.org"
    }
}
```

To run the UI locally:

```bash
streamlit run ui_app.py
```

To install dependencies first:

```bash
pip install -r requirements.txt
```

---

## Example End-to-End Flow
A typical full workflow looks like this:

1. The user fills out the Streamlit form in `ui_app.py`
2. The UI builds a payload and calls `run_task("flyer", payload)`
3. `run_task.py` creates a LangGraph pipeline with nodes from `app.py`
4. The email generation node drafts a proposal email
5. The form generation node creates a registration schema and QR code
6. The flyer generation node creates and selects a flyer design
7. The generated files are returned to the UI

This shows how one input can power multiple related outputs through a coordinated AI workflow.

---

## Error Handling and Robustness
The project includes error handling and fallback logic:
- date parsing uses `try/except` and falls back safely if parsing fails
- form generation includes sensible defaults for missing event fields
- `run_task.py` wraps execution in a try/except block and returns an error message instead of crashing
- `form_app.py` falls back to a default form if the generated JSON file cannot be loaded
